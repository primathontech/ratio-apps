# ratio-apps — Consolidation & Agentic-System Design

**Date:** 2026-06-17
**Status:** Approved design → implementation plan pending
**Author:** Prince Balhara (with Claude)

## Problem

Three separate repos share the same `ratio-app-boilerplate` pnpm monorepo and
differ only in their vendor modules:

| Source repo            | Backend modules            | Admin SPAs                       |
|------------------------|----------------------------|----------------------------------|
| `anayltics-ratio-app`  | `google` (+ `_template`)   | `admin-google`                   |
| `meta-g4_ratio_app`    | `meta` (+ `_template`)     | `admin-meta`                     |
| `posthog`              | `posthog`, `moengage`      | `admin-posthog`, `admin-moengage`|

Maintaining three copies of the same `core/` and skills library is wasteful and
the copies have drifted. We want **one repo, `ratio-apps`**, holding all vendor
functionality on a single shared core — which is exactly the grain the
boilerplate was designed for ("one process, many modules"; `posthog` already
runs two modules via `APPS = ['posthog','moengage']`).

## Decisions (confirmed)

1. **Unified backend** — one `apps/backend` with all four vendor modules on one
   shared core; all admin SPAs side-by-side. (Not preserved sub-projects.)
2. **Core merge = newest-per-file, flag conflicts** — take the most complete
   version of each diverged core file; surface genuine meta-vs-posthog conflicts
   for a human decision instead of silently picking.
3. **New sibling folder `ratio-apps` with fresh git** (no stitched history).
   Source repos left untouched.
4. **Single spec covers both phases** (consolidation + agentic-system), with
   Phase 2 specifics verified against the merged structure before execution.

## Target structure

```
ratio-apps/                          # new git repo, base = anayltics-ratio-app
  apps/
    backend/                         # APPS = ['google','meta','posthog','moengage']
      src/
        core/                        # single reconciled shared infrastructure
        modules/{_template,google,meta,posthog,moengage}/
        config/env.schema.ts         # derives RATIO_<SLUG>_* keys from APPS
    _template-admin/                 # scaffolder copy-source (kept, unwired)
    admin-google/
    admin-meta/
    admin-posthog/
    admin-moengage/
  packages/shared/                   # union of all vendors' Zod schemas + events
  .agents/skills/                    # full 18-skill set (symlinked .claude/skills)
  docs/agent/apps/{google,meta,posthog,moengage}/
  docker-compose.yml                 # one MySQL, four databases
  ecosystem.config.cjs
  package.json                       # merged, per-vendor namespaced scripts
  pnpm-lock.yaml                     # regenerated by single pnpm install
```

## Phase 1 — Consolidation

### 1.1 Base
Copy `anayltics-ratio-app` wholesale into `ratio-apps` (it carries the fullest
skills library and tooling). This becomes the canonical core + scaffolding.

### 1.2 Vendor modules (self-contained, copy verbatim)
- `modules/meta/` ← from `meta-g4_ratio_app`
- `modules/posthog/`, `modules/moengage/` ← from `posthog`

Each module owns its own DB, env keys (`RATIO_<SLUG>_*`), controllers, services,
bootstrap + webhook handler classes. No cross-module coupling.

### 1.3 Wiring (the repo's own deterministic recipe — see `vendor-scaffolder`)
1. `apps/backend/src/modules/apps.ts` → `APPS = ['google','meta','posthog','moengage'] as const`.
   Keep the slug regex permissive enough for `_template` only where needed;
   real slugs are `[a-z0-9-]`.
2. `apps/backend/src/app.module.ts` → register all four concrete `<App>Module`s
   in `REGISTERED_MODULES` map **and** `imports[]` (load-time assertion enforces).
3. `config/env.schema.ts` derives `RATIO_<SLUG>_*` keys from `APPS` automatically
   — no manual edit beyond confirming the list.

### 1.4 Core reconciliation (newest-per-file)
Diverged files to reconcile (analytics is base; compare against meta + posthog):

- `core/oauth/oauth.service.ts`
- `core/webhooks/webhooks.service.ts`
- `core/factories/app-module.factory.ts`
- `core/common/pipes/zod-validation.pipe.ts`
- `core/common/resolve-pixel-path.ts`
- `core/common/decorators/merchant.decorator.ts`
- `core/common/safe-inline-json.ts`
- `core/db/kysely-factory.ts`
- `core/db/shared-migrations.ts`

Procedure per file: three-way compare; take the superset/newest that satisfies
all consuming modules. meta's core carries newer event-bus / catalog-sync work;
posthog's carries db/decorator tweaks. Where meta's and posthog's edits to the
same hunk genuinely conflict, **stop and present the conflict** for a decision —
do not auto-pick. Most diffs are additive evolutions of the same base and should
union cleanly.

### 1.5 packages/shared
Union the per-vendor schema + event-constant files. Vendor files are namespaced
(e.g. `meta.*`, `posthog.*`) so no collisions are expected; confirm exports
barrel cleanly.

### 1.6 Frontends
Copy `admin-meta`, `admin-posthog`, `admin-moengage` as-is. Independent Vite +
TanStack Router SPAs; no shared state to reconcile.

### 1.7 Root config
- `package.json` — merge `dev:`/`build:`/`migrate:`/`deploy:` scripts (already
  per-vendor namespaced in each source). Keep posthog's S3/CloudFront deploy
  scripts for its admins; keep analytics/meta single-artifact deploy.
- `docker-compose.yml` — one MySQL service provisioning four databases (one per
  module, per the per-module-DB rule); backend serves built admins.
- `ecosystem.config.cjs` — merge PM2 app entries.
- `.env.example` — union of all `RATIO_<SLUG>_*` blocks + shared keys.
- `pnpm-workspace.yaml` — `apps/*`, `packages/*`, exclude `apps/_template-admin`.

### 1.8 docs/agent
Carry each vendor's `docs/agent/apps/<slug>/{CONTEXT,PRD,TRD,TDD}.md` and
`STATE.json`; merge `FEATURES.md` (union of capabilities) and `context/INDEX.md`.

### 1.9 Phase 1 Definition of Done
`pnpm install && pnpm verify` (lint + typecheck + test + build) green across all
four modules and four admin SPAs. Initial commit on the fresh `ratio-apps` repo.

## Phase 2 — Agentic-system improvement

### 2.1 Baseline consolidation
The skills library has drifted hard:
- `anayltics-ratio-app` — full 18 skills (incl. `build-app`, `execute`,
  `remember`, `write-plan`).
- `meta-g4_ratio_app` — 13 skills (missing the four above).
- `posthog` — effectively none.

Phase 1 already seeds `ratio-apps` with analytics' full set as the baseline.

### 2.2 Improvements (verified against merged structure before execution)
- **`vendor-scaffolder`** — make it explicitly multi-vendor-aware: the copy-rename
  recipe must append to a 4+ entry `APPS` list and `REGISTERED_MODULES` map
  without disturbing siblings; assert no collision with existing slugs.
- **`house-conventions`** — codify the now-real multi-module invariants (per-module
  DB, `RATIO_<SLUG>_*` namespacing, `core/` extend-not-fork) with the merged repo
  as the worked example.
- **`build-app` / `context-keeper` / `remember`** — ensure per-app context files
  and `FEATURES.md` updates scope to the correct `<slug>` now that four coexist.
- **`AGENTS.md`** — update the "Add a new app" and stack sections to reflect the
  unified repo (four live vendors, not one).

Concrete edits are finalized in the implementation plan after the merged
structure exists, so skills are designed against real files, not assumptions.

### 2.3 Phase 2 Definition of Done
Skills + `AGENTS.md` reflect the unified multi-vendor repo; a dry-run of
`vendor-scaffolder` (or its documented recipe) against a throwaway slug wires
cleanly and `pnpm verify` stays green; changes recorded via `remember`.

## Risks & mitigations

- **Core conflict (meta vs posthog edits to same hunk):** flag-and-decide, never
  auto-pick — codified in 1.4.
- **Hidden core→module coupling:** a module may depend on a core change not in
  analytics' base; `pnpm -r typecheck` after wiring catches missing symbols.
- **DB name collisions:** each module already namespaces its DB; verify the four
  names are distinct in docker-compose + each module's config.
- **Dependency version skew across the three lockfiles:** single `pnpm install`
  resolves to one tree; `pnpm verify` proves it builds.

## Out of scope
- Rewriting any vendor's business logic.
- Changing deploy targets (keep each vendor's existing deploy story).
- Stitching git history from the source repos.
